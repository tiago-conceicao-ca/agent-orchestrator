This quickstart walks through the smallest useful CAHI loop: start the dashboard, create one worker session, watch it work, and clean it up after the PR is merged.

<Callout type="info" title="Before you start">
  Complete [Installation](/docs/installation) first. You need `cahi`, Git, one authenticated source-control CLI such as `gh`, and one signed-in agent CLI.
</Callout>

## Pick A Safe First Task

Use a repository where you can push a branch and open a pull request. For the first run, choose a task that is easy to review:

- Fix a small bug with a clear failure.
- Update a short documentation page.
- Add a narrow test.
- Make a small refactor with obvious acceptance criteria.

Avoid broad tasks like “improve auth” or “clean up the app.” CAHI can run many agents, but each worker still needs a task with a visible finish line.

## Run The First Session

<Steps>
<Step>

### Start CAHI

From the repository you want CAHI to manage:

```bash
cd ~/code/my-repo
cahi start
```

On the first run, CAHI creates `cahi.yaml`, starts the dashboard, and starts an orchestrator session for the project. The dashboard URL is printed in the terminal, usually `http://localhost:4000`.

Keep this terminal running. It owns the dashboard and lifecycle polling.

</Step>
<Step>

### Spawn one worker

Open a second terminal in the same repository.

<Tabs items={["GitHub issue", "Freeform prompt"]}>
  <Tab value="GitHub issue">
  ```bash
  cahi spawn 42
  ```

  Replace `42` with the issue number. CAHI fetches the issue through `gh`, creates a worktree, starts the configured worker agent, and gives it the issue context.
  </Tab>
  <Tab value="Freeform prompt">
  ```bash
  cahi spawn --prompt "Update the README install section to mention Node 20"
  ```

  Use this when the task is not tracked in GitHub, GitLab, or Linear yet. Keep the prompt specific and reviewable.
  </Tab>
</Tabs>

The command prints the session id and dashboard URL. Session names use the project prefix, for example `myrepo-1`.

</Step>
<Step>

### Watch the dashboard

Open the session card. You should see:

- The worker activity state: active, ready, idle, waiting for input, blocked, or exited.
- The worktree path and branch name.
- The live terminal output.
- The PR link after the agent creates one.
- CI and review state after a PR exists.

Use `cahi status` when you want the same high-level view in a terminal:

```bash
cahi status
```

</Step>
<Step>

### Intervene only when needed

CAHI routes routine feedback back to the worker:

- CI failure: CAHI sends failure context to the session.
- Requested changes: CAHI sends the review feedback to the session.
- Merge conflicts: CAHI can ask the worker to rebase or resolve the conflict.
- Agent stuck or waiting for input: CAHI notifies you.

If you need to give the worker a direct instruction, use `cahi send`:

```bash
cahi send myrepo-1 "Keep the fix smaller. Do not refactor the API layer."
```

Use `cahi send` instead of raw terminal input. It preserves CAHI's busy detection, retry handling, and message formatting.

</Step>
<Step>

### Review and merge

When the PR is green and ready, review it like any other pull request. CAHI does not merge by default.

After the PR is merged or the issue is closed, clean up completed sessions:

```bash
cahi session cleanup --dry-run
cahi session cleanup
```

</Step>
</Steps>

## What CAHI Created

| Item | What it means |
| --- | --- |
| `cahi.yaml` | Project config: plugins, projects, reactions, runtime, and notifier choices. |
| Orchestrator session | A coordinating session started by `cahi start`; it supervises worker sessions. |
| Worker session | The agent process that works on one issue or prompt. |
| Worktree | An isolated checkout for the worker's branch. |
| Session metadata | Files under `~/.cahi/...` that let CAHI track branch, PR, status, and runtime state. |

## If Something Looks Wrong

| Symptom | First check |
| --- | --- |
| Dashboard is not updating | Make sure the `cahi start` terminal is still running. |
| `cahi spawn` warns that CAHI is not running | Start CAHI with `cahi start` before spawning. |
| GitHub issue or PR data is missing | Run `gh auth status` and check the `repo` field in `cahi.yaml`. |
| Agent started but does nothing | Open the session terminal and send a clear instruction with `cahi send`. |
| Windows spawn fails with tmux errors | Set `defaults.runtime: process` in `cahi.yaml`. |
